### Writing Style Guidelines

1. **Avoid jargon**: Explain technical concepts when they appear
   - BAD: "MoE architectures with sparse attention enable efficient scaling"
   - GOOD: "Using specialized sub-models that activate selectively (mixture-of-experts) allows systems to scale efficiently without proportional compute increases"

2. **Lead with "why it matters"**:
   - BAD: "GraphRAG combines knowledge graphs with retrieval"
   - GOOD: "Connecting retrieved information through knowledge graphs helps AI systems reason about relationships between facts, improving accuracy on complex questions by 20-30%"

3. **Include concrete evidence**: Pair recommendations with specific results
   - Include example papers for each recommendation
   - Quote improvements in understandable terms (e.g., "20% more accurate" rather than just "0.87 F1")

### Paper Links

All paper references should use this format:
- Link URL: `https://papers.lunadong.com/paper/{paper_id}`
- Example: Paper ID 52 → `https://papers.lunadong.com/paper/52`

### Field Guidelines

| Field | Count | Description |
|-------|-------|-------------|
| `area_overview.definition` | Max 200 characters | A single concise sentence. Do NOT write a full paragraph. |
| `area_overview.key_paradigms` | Use taxonomy categories only | Each paradigm must correspond to a category from the taxonomy. Do NOT invent new paradigms beyond what the taxonomy defines. |
| `cross_topic_connections` | One per adjacent topic pair | You MUST produce a transition for EVERY consecutive pair of topics in the order they appear. If there are N topics, produce exactly N-1 connection entries covering every adjacent pair. |
| `field_timeline.periods[].key_developments` | 2-4 per period | Each development MUST embed a markdown paper link: `[ShortName](https://papers.lunadong.com/paper/{paper_id})`. Keep each development to 1-2 sentences. |
| `practical_recommendations` | 6-8 | Actionable recommendations with priority levels. Include 1-2 example papers per recommendation. |
| `key_takeaways` | 4-6 | High-level insights with emoji, title, description, AND `one_liner`. Write for a broad technical audience. |
| `emerging_trends` | 3-5 | Trends with supporting evidence and 2-3 example papers each. Use `supporting_papers` (not `example_papers`) as the field name. |
| `research_opportunities` | 4-6 | Gaps and opportunities with difficulty and impact assessment |
| `benchmark_leaderboard` | 3-5 | Key benchmarks from across all topics. Include top 3-5 results per benchmark with paper links. |
| `glossary` | 15-25 | Technical terms used across topics, with plain-language definitions. Merge and deduplicate from individual topic glossaries. |

### Priority Levels

- **High**: Critical recommendations backed by multiple high-impact papers
- **Medium**: Important recommendations for near-term consideration
- **Low**: Promising directions with preliminary evidence

### Focus Areas

- Cross-cutting patterns that span multiple topics
- Practical implications for real-world systems
- Gaps and opportunities for future research
- Connections between different topic areas
